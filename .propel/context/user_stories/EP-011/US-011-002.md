# User Story: ARIA Labels and Screen Reader Support

**Epic:** EP-011 - Accessibility & WCAG Compliance  
**Story ID:** US-011-002  
**Priority:** P0 - Critical  
**Estimate:** 5 Story Points (2 days)

---

## User Story

**As a** screen reader user  
**I want** proper ARIA labels and semantic HTML throughout the application  
**So that** I can understand and navigate the interface using assistive technologies

---

## Acceptance Criteria

### AC1: Semantic HTML Structure
**Given** I navigate the application with a screen reader  
**When** I explore the page structure  
**Then**:
- Proper heading hierarchy (h1 > h2 > h3 > h4, no skipped levels)
- Landmark regions: `<header>`, `<nav>`, `<main>`, `<footer>`, `<aside>`
- Lists use `<ul>`, `<ol>`, `<li>` elements
- Tables use `<table>`, `<thead>`, `<tbody>`, `<th>`, `<td>`
- Forms use `<form>`, `<label>`, `<input>`, `<select>`, `<textarea>`
- Buttons use `<button>` not `<div>` with click handlers

### AC2: ARIA Labels for Icon Buttons
**Given** I encounter icon-only buttons  
**When** I focus on them  
**Then**:
- All icon buttons have `aria-label` attribute
- Labels are descriptive and actionable
- Examples:
  - `aria-label="Upload document"`
  - `aria-label="Delete document"`
  - `aria-label="Close modal"`
  - `aria-label="Next page"`
- Labels match visual purpose
- No generic labels like "Button" or "Icon"

### AC3: Form Labels and Associations
**Given** I navigate a form  
**When** I focus on form inputs  
**Then**:
- All inputs have associated `<label>` elements
- Labels use `for` attribute matching input `id`
- Or use `aria-labelledby` for complex associations
- Required fields indicated with `aria-required="true"`
- Invalid fields have `aria-invalid="true"`
- Error messages associated with `aria-describedby`

### AC4: ARIA Live Regions
**Given** dynamic content updates  
**When** changes occur  
**Then**:
- Toast notifications use `aria-live="polite"` or `aria-live="assertive"`
- Error messages use `aria-live="assertive"`
- Success messages use `aria-live="polite"`
- Loading states announced with `aria-busy="true"`
- Status updates announced automatically
- Announcements don't interrupt current reading

### AC5: Modal Dialogs
**Given** I open a modal dialog  
**When** the modal appears  
**Then**:
- Modal has `role="dialog"` or `role="alertdialog"`
- Modal has `aria-modal="true"`
- Modal has `aria-labelledby` pointing to title
- Modal has `aria-describedby` pointing to description (if applicable)
- Focus moves to modal on open
- Background content has `aria-hidden="true"`

### AC6: Navigation Menus
**Given** I navigate menus  
**When** I use the navigation  
**Then**:
- Navigation uses `<nav>` element
- Nav has `aria-label` to distinguish multiple navs
- Current page indicated with `aria-current="page"`
- Dropdown menus use `aria-expanded` and `aria-haspopup`
- Menu items use proper list structure
- Breadcrumbs use `<nav aria-label="Breadcrumb">`

### AC7: Data Tables
**Given** I navigate data tables  
**When** I explore table content  
**Then**:
- Tables have `<caption>` or `aria-label`
- Headers use `<th>` with `scope="col"` or `scope="row"`
- Complex tables use `aria-labelledby` and `aria-describedby`
- Sortable columns indicate sort state with `aria-sort`
- Interactive cells have proper roles
- Empty cells have content or are marked appropriately

### AC8: Image Alternative Text
**Given** I encounter images  
**When** images are present  
**Then**:
- All images have `alt` attribute
- Decorative images have `alt=""` (empty)
- Informative images have descriptive alt text
- Complex images have detailed descriptions
- Icons have `aria-label` or are hidden with `aria-hidden="true"`
- SVGs have `<title>` and `<desc>` elements

### AC9: Loading and Progress Indicators
**Given** content is loading  
**When** I wait for operations  
**Then**:
- Loading spinners have `aria-label="Loading"`
- Progress bars have `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`
- Loading states announced with `aria-busy="true"`
- Completion announced via live region
- Skeleton screens have appropriate labels

### AC10: Custom Components
**Given** I use custom interactive components  
**When** standard HTML elements aren't used  
**Then**:
- Custom components have appropriate ARIA roles
- State changes communicated via ARIA attributes
- Examples:
  - Toggle: `role="switch"`, `aria-checked`
  - Tabs: `role="tablist"`, `role="tab"`, `role="tabpanel"`
  - Accordion: `aria-expanded`, `aria-controls`
  - Combobox: `role="combobox"`, `aria-autocomplete`
- Keyboard interactions match ARIA patterns

## Technical Details

### ARIA Label Examples
```typescript
// Icon Button
<button aria-label="Upload document">
  <Upload className="h-4 w-4" />
</button>

// Close Button
<button aria-label="Close modal" onClick={onClose}>
  <X className="h-4 w-4" />
</button>

// Navigation
<nav aria-label="Main navigation">
  <ul>
    <li><a href="/" aria-current="page">Dashboard</a></li>
    <li><a href="/upload">Upload</a></li>
  </ul>
</nav>

// Breadcrumb
<nav aria-label="Breadcrumb">
  <ol>
    <li><a href="/">Home</a></li>
    <li aria-current="page">Upload Document</li>
  </ol>
</nav>
```

### Form Labels
```typescript
// Label with for attribute
<label htmlFor="email">Email Address</label>
<input
  id="email"
  type="email"
  aria-required="true"
  aria-invalid={hasError}
  aria-describedby={hasError ? "email-error" : undefined}
/>
{hasError && (
  <span id="email-error" className="text-error">
    Please enter a valid email address
  </span>
)}

// Fieldset for radio groups
<fieldset>
  <legend>Choose LLM Model</legend>
  <label>
    <input type="radio" name="model" value="openai" />
    OpenAI GPT
  </label>
  <label>
    <input type="radio" name="model" value="gemini" />
    Google Gemini
  </label>
</fieldset>
```

### ARIA Live Regions
```typescript
// Toast Component
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
  className="toast"
>
  {message}
</div>

// Error Toast
<div
  role="alert"
  aria-live="assertive"
  aria-atomic="true"
  className="toast-error"
>
  {errorMessage}
</div>

// Loading State
<div aria-live="polite" aria-busy="true">
  Loading documents...
</div>
```

### Modal Dialog
```typescript
<div
  role="dialog"
  aria-modal="true"
  aria-labelledby="modal-title"
  aria-describedby="modal-description"
>
  <h2 id="modal-title">Delete Document</h2>
  <p id="modal-description">
    Are you sure you want to delete this document? This action cannot be undone.
  </p>
  <button onClick={onConfirm}>Delete</button>
  <button onClick={onCancel}>Cancel</button>
</div>

// Background overlay
<div aria-hidden="true" className="modal-overlay" />
```

### Data Table
```typescript
<table aria-label="Document history">
  <caption className="sr-only">List of processed documents</caption>
  <thead>
    <tr>
      <th scope="col">Filename</th>
      <th scope="col" aria-sort="descending">Upload Date</th>
      <th scope="col">Status</th>
      <th scope="col">Actions</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>document.pdf</td>
      <td>Jan 15, 2026</td>
      <td>
        <span className="badge-success" role="status">Success</span>
      </td>
      <td>
        <button aria-label="View document.pdf">View</button>
        <button aria-label="Delete document.pdf">Delete</button>
      </td>
    </tr>
  </tbody>
</table>
```

### Custom Toggle Component
```typescript
<button
  role="switch"
  aria-checked={isChecked}
  aria-label="Toggle card view"
  onClick={() => setIsChecked(!isChecked)}
>
  <span aria-hidden="true">{isChecked ? 'On' : 'Off'}</span>
</button>
```

### Progress Bar
```typescript
<div
  role="progressbar"
  aria-valuenow={progress}
  aria-valuemin={0}
  aria-valuemax={100}
  aria-label="Upload progress"
>
  <div style={{ width: `${progress}%` }} />
</div>
```
