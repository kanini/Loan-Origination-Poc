# User Story: Unit Testing with Vitest and React Testing Library

**Epic:** EP-015 - Testing & Quality Assurance  
**Story ID:** US-015-001  
**Priority:** P1 - High  
**Estimate:** 8 Story Points (3-4 days)

---

## User Story

**As a** Frontend Developer  
**I want** to write unit tests for React components using Vitest and React Testing Library  
**So that** I can ensure component functionality, prevent regressions, and maintain code quality

---

## Acceptance Criteria

### AC1: Vitest Installation and Configuration
**Given** the React project is set up  
**When** I install Vitest  
**Then**:
- Vitest v1.2+ installed as dev dependency
- @testing-library/react v14+ installed
- @testing-library/jest-dom v6+ installed for custom matchers
- @testing-library/user-event v14+ installed for user interactions
- happy-dom or jsdom installed as test environment
- Vitest configuration file created

### AC2: Test Environment Setup
**Given** Vitest is installed  
**When** I configure the test environment  
**Then**:
- Test environment set to happy-dom (faster) or jsdom
- Global test utilities configured
- Custom matchers from jest-dom available
- Test setup file runs before each test
- TypeScript types for testing libraries available

### AC3: Test Scripts
**Given** Vitest is configured  
**When** I add test scripts to package.json  
**Then**:
- `npm run test` runs tests in watch mode
- `npm run test:ui` opens Vitest UI
- `npm run test:run` runs tests once (CI mode)
- `npm run test:coverage` generates coverage report
- Scripts work correctly in all environments

### AC4: Component Testing Coverage
**Given** I write unit tests  
**When** I test React components  
**Then** tests cover:
- Button component (all variants, states, interactions)
- Input component (validation, error states, keyboard input)
- Card component (rendering, hover effects)
- Modal component (open/close, keyboard navigation, focus trap)
- Toast component (auto-dismiss, manual dismiss)
- Form components (validation, submission, error handling)
- Navigation components (active states, routing)

### AC5: Test Structure and Organization
**Given** I organize test files  
**When** I create tests  
**Then**:
- Test files colocated with components: `Component.test.tsx`
- Test files in `__tests__` folder (alternative)
- Consistent test structure: describe → it → arrange/act/assert
- Clear test descriptions following pattern: "should [expected behavior] when [condition]"
- Setup and teardown in beforeEach/afterEach hooks

### AC6: Rendering Tests
**Given** I test component rendering  
**When** I write render tests  
**Then**:
- Components render without errors
- Correct text content displayed
- Correct elements present in DOM
- Props passed correctly to child components
- Conditional rendering works as expected
- Loading states render correctly

### AC7: Interaction Tests
**Given** I test user interactions  
**When** I write interaction tests  
**Then**:
- Button clicks trigger correct callbacks
- Form inputs update on user typing
- Form submission calls correct handlers
- Keyboard navigation works (Tab, Enter, Esc)
- Hover effects apply correct styles
- Focus management works correctly

### AC8: Accessibility Tests
**Given** I test accessibility  
**When** I write a11y tests  
**Then**:
- ARIA labels present and correct
- Semantic HTML used (button, nav, main, etc.)
- Focus indicators visible
- Screen reader text available
- Keyboard navigation functional
- Color contrast meets WCAG standards (manual check)

### AC9: Async Behavior Tests
**Given** I test async operations  
**When** I write async tests  
**Then**:
- Loading states display during data fetching
- Success states display after data loads
- Error states display on failure
- Retry functionality works
- Optimistic updates work correctly
- React Query integration tested with MSW (Mock Service Worker)

### AC10: Coverage Requirements
**Given** tests are written  
**When** I run coverage report  
**Then**:
- Overall coverage ≥80%
- Statement coverage ≥80%
- Branch coverage ≥75%
- Function coverage ≥80%
- Line coverage ≥80%
- Critical components have ≥90% coverage

### Test Setup File
```typescript
// src/test/setup.ts
import { expect, afterEach } from 'vitest'
import { cleanup } from '@testing-library/react'
import * as matchers from '@testing-library/jest-dom/matchers'

// Extend Vitest's expect with jest-dom matchers
expect.extend(matchers)

// Cleanup after each test
afterEach(() => {
  cleanup()
})

// Mock window.matchMedia
Object.defineProperty(window, 'matchMedia', {
  writable: true,
  value: vi.fn().mockImplementation(query => ({
    matches: false,
    media: query,
    onchange: null,
    addListener: vi.fn(),
    removeListener: vi.fn(),
    addEventListener: vi.fn(),
    removeEventListener: vi.fn(),
    dispatchEvent: vi.fn(),
  })),
})
```

### Example Component Test
```typescript
// src/components/ui/button.test.tsx
import { describe, it, expect, vi } from 'vitest'
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { Button } from './button'

describe('Button', () => {
  it('should render with children text', () => {
    render(<Button>Click me</Button>)
    expect(screen.getByRole('button', { name: /click me/i })).toBeInTheDocument()
  })

  it('should call onClick handler when clicked', async () => {
    const handleClick = vi.fn()
    const user = userEvent.setup()
    
    render(<Button onClick={handleClick}>Click me</Button>)
    
    await user.click(screen.getByRole('button'))
    expect(handleClick).toHaveBeenCalledTimes(1)
  })

  it('should be disabled when disabled prop is true', () => {
    render(<Button disabled>Click me</Button>)
    expect(screen.getByRole('button')).toBeDisabled()
  })

  it('should show loading spinner when loading prop is true', () => {
    render(<Button loading>Click me</Button>)
    expect(screen.getByRole('button')).toBeDisabled()
    expect(screen.getByTestId('spinner')).toBeInTheDocument()
  })

  it('should apply primary variant styles', () => {
    render(<Button variant="primary">Click me</Button>)
    const button = screen.getByRole('button')
    expect(button).toHaveClass('bg-primary')
  })

  it('should be keyboard accessible', async () => {
    const handleClick = vi.fn()
    const user = userEvent.setup()
    
    render(<Button onClick={handleClick}>Click me</Button>)
    
    const button = screen.getByRole('button')
    button.focus()
    expect(button).toHaveFocus()
    
    await user.keyboard('{Enter}')
    expect(handleClick).toHaveBeenCalledTimes(1)
  })
})
```

### Example Async Test with React Query
```typescript
// src/pages/Dashboard.test.tsx
import { describe, it, expect, beforeEach } from 'vitest'
import { render, screen, waitFor } from '@testing-library/react'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'
import { BrowserRouter } from 'react-router-dom'
import { setupServer } from 'msw/node'
import { rest } from 'msw'
import Dashboard from './Dashboard'

const server = setupServer(
  rest.get('/api/documents/recent/', (req, res, ctx) => {
    return res(ctx.json({
      documents: [
        { id: '1', filename: 'test.pdf', status: 'success' },
      ],
    }))
  })
)

beforeAll(() => server.listen())
afterEach(() => server.resetHandlers())
afterAll(() => server.close())

function renderWithProviders(ui: React.ReactElement) {
  const queryClient = new QueryClient({
    defaultOptions: {
      queries: { retry: false },
    },
  })

  return render(
    <QueryClientProvider client={queryClient}>
      <BrowserRouter>
        {ui}
      </BrowserRouter>
    </QueryClientProvider>
  )
}

describe('Dashboard', () => {
  it('should display loading state initially', () => {
    renderWithProviders(<Dashboard />)
    expect(screen.getByTestId('skeleton')).toBeInTheDocument()
  })

  it('should display recent documents after loading', async () => {
    renderWithProviders(<Dashboard />)
    
    await waitFor(() => {
      expect(screen.getByText('test.pdf')).toBeInTheDocument()
    })
  })

  it('should display error state on API failure', async () => {
    server.use(
      rest.get('/api/documents/recent/', (req, res, ctx) => {
        return res(ctx.status(500))
      })
    )

    renderWithProviders(<Dashboard />)
    
    await waitFor(() => {
      expect(screen.getByText(/failed to load/i)).toBeInTheDocument()
    })
  })
})
```

## Notes

- Vitest is Vite-native and faster than Jest
- Use happy-dom for faster tests (jsdom for better compatibility)
- Mock Service Worker (MSW) for API mocking
- Test user behavior, not implementation details
- Follow Testing Library best practices (query by role, not testId)
- Run tests in watch mode during development
- Generate coverage reports before each release
- Add pre-commit hook to run tests (optional)
